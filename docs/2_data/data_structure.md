# Data Structure

Based on the business and non-functional requirements, the following entity-relationship data structure is proposed for the AI-Powered Personalized News Aggregator.

## 1. Entities & Fields

### 1.1 User
Represents an individual user in the system.
- `id` (UUID, Primary Key)
- `email` (String, Unique)
- `oauth_provider` (Enum: `google`, `github`)
- `oauth_id` (String)
- `interests` (Array of Strings)
- `language_preferences` (Array of Strings)
- `created_at` (Timestamp)
- `updated_at` (Timestamp)

### 1.2 DeliveryChannel
Represents the communication channels a user has connected.
- `id` (UUID, Primary Key)
- `user_id` (UUID, Foreign Key -> User.id)
- `type` (Enum: `in-app`, `email`, `telegram`, `slack`)
- `config` (JSON) - Stores channel-specific configuration (e.g., Slack webhook URL, Telegram chat ID, verified email address). *Note: Must be encrypted at rest.*
- `created_at` (Timestamp)
- `updated_at` (Timestamp)

### 1.3 GlobalSource
Represents a unique news source (RSS, Atom, or Web URL). Cached globally to avoid redundant fetching.
- `id` (UUID, Primary Key)
- `url` (String, Unique)
- `type` (Enum: `rss`, `atom`, `web`)
- `status` (Enum: `active`, `paused`) - Transitions to paused if failed 5 times.
- `failed_fetch_count` (Integer) - Tracks consecutive failures.
- `last_fetched_at` (Timestamp)
- `recent_items` (JSON Array) - Persistent, per-source bounded FIFO history sized to hold at least 150 seen articles, with each entry containing a URL hash and, when available, a GUID hash. It supports deduplication after the corresponding `IngestedArticle` rows have been purged, including for items without a publication timestamp.
- `created_at` (Timestamp)

### 1.4 ProcessingFlow
Represents an automated rule/flow configured by a user. Max 5 per user.
- `id` (UUID, Primary Key)
- `user_id` (UUID, Foreign Key -> User.id)
- `name` (String)
- `frequency` (Enum: `daily`)
- `ai_model` (Enum: `gpt-5.4-mini`)
- `prompt_type` (Enum: `predefined`, `custom`)
- `prompt_template` (Text) - Encrypted if custom.
- `created_at` (Timestamp)
- `updated_at` (Timestamp)

### 1.5 FlowSource (Join Table)
Maps a processing flow to one or more global sources.
- `flow_id` (UUID, Foreign Key -> ProcessingFlow.id)
- `source_id` (UUID, Foreign Key -> GlobalSource.id)

### 1.6 FlowDeliveryChannel (Join Table)
Maps a processing flow to one or more delivery channels.
- `flow_id` (UUID, Foreign Key -> ProcessingFlow.id)
- `channel_id` (UUID, Foreign Key -> DeliveryChannel.id)

### 1.7 IngestedArticle
Stores the raw extracted content from a GlobalSource. Used by all flows that subscribe to the source.
- `id` (UUID, Primary Key)
- `source_id` (UUID, Foreign Key -> GlobalSource.id)
- `external_guid` (String, Nullable) - Unique identifier from the feed (e.g. RSS `<guid>` or Atom `<id>`).
- `title` (String)
- `url` (String)
- `content` (Text) - Main text extracted from the URL, stripped of ads/menus.
- `published_at` (Timestamp, Nullable)
- `created_at` (Timestamp)
- *Constraints:* To reliably prevent duplicate ingestion of the same article on successive polling cycles while supporting feeds that reuse URLs across distinct items (e.g. podcast episodes or blog updates):
  1. A partial unique index on `(source_id, external_guid)` filtered by `WHERE external_guid IS NOT NULL`.
  2. A partial unique index on `(source_id, url)` filtered by `WHERE external_guid IS NULL`. This fallback URL deduplication only triggers when no GUID is provided by the feed.

### 1.8 ProcessedDigest
Stores the final processed content generated by the AI for a specific flow. Retained for 1 week.
- `id` (UUID, Primary Key)
- `flow_id` (UUID, Foreign Key -> ProcessingFlow.id)
- `content` (Text) - The summarized digest.
- `token_usage` (Integer) - Tracks AI tokens consumed.
- `user_feedback` (Enum: `thumbs_up`, `thumbs_down`, `none`)
- `created_at` (Timestamp) - Used by cleanup jobs for 7-day retention deletion.

### 1.9 DigestDeliveryAttempt
Tracks the delivery state and history for each delivery channel associated with a digest. Enforces idempotent delivery.
- `id` (UUID, Primary Key)
- `digest_id` (UUID, Foreign Key -> ProcessedDigest.id ON DELETE CASCADE)
- `channel_id` (UUID, Nullable, Foreign Key -> DeliveryChannel.id ON DELETE SET NULL)
- `status` (Enum: `pending`, `sending`, `delivered`, `failed`)
- `error_message` (Text, Nullable) - Details of the failure, if any.
- `attempted_at` (Timestamp, Nullable)
- `locked_at` (Timestamp, Nullable) - Timestamp when status was set to `sending`, enabling crash-recovery checks.
- `retry_count` (Integer, Default: 0) - Tracks the number of retries for exponential backoff.
- `next_attempt_at` (Timestamp, Nullable) - Timestamp when the attempt is next eligible for retry after backoff delay.
- `created_at` (Timestamp)
- *Constraint:* Unique composite index on `(digest_id, channel_id)` to prevent duplicate rows in the database.
- *Idempotency & Delivery Protocol:* To satisfy the best-effort idempotent delivery NFR for channels that lack native API-level deduplication (such as Slack Webhooks, the Telegram Bot API, and standard SMTP mail delivery), the background worker employs a **Pre-flight State Locking** protocol:
  1. Before executing the external API call, the worker initiates a database transaction, locks the attempt row using `SELECT FOR UPDATE`, verifies the status is `pending` OR `failed` (where the retry is eligible based on `next_attempt_at <= NOW()`), and updates it to `sending` with `locked_at = NOW()`.
  2. To prevent messages from being permanently lost if a worker crashes *before* initiating the external request, the scheduler runs a recovery sweep. Any attempt in `sending` status with `locked_at` older than a lock lease timeout (e.g., 5 minutes) is considered safely recoverable and reset back to `pending` for retry. To prevent this recovery sweep from racing active slow deliveries, the worker's network timeouts must be strictly bounded: HTTP request timeouts (for Slack and Telegram) and SMTP connection, greeting, and socket timeouts (for email) must be configured to a maximum of 30 seconds, guaranteeing a worker never holds the lock for longer than 30 seconds unless it has crashed.
  3. If a crash occurs *after* the external API accepts the message but *before* the success transaction is committed, a retry may result in a duplicate. Since these external channels do not provide deduplication, this at-least-once transmission is a known technical limitation, mitigated by minimizing the time window between the API response and the status commit.

## 2. Relationships Overview
- **User** 1 : M **DeliveryChannel**
- **User** 1 : M **ProcessingFlow** (Max 5)
- **ProcessingFlow** M : N **GlobalSource** (via FlowSource)
- **ProcessingFlow** M : N **DeliveryChannel** (via FlowDeliveryChannel)
- **GlobalSource** 1 : M **IngestedArticle**
- **ProcessingFlow** 1 : M **ProcessedDigest**
- **ProcessedDigest** 1 : M **DigestDeliveryAttempt**
- **DeliveryChannel** 1 : M **DigestDeliveryAttempt**

## 3. Storage Constraints & Policies
- **Data Isolation:** User's custom prompts and config JSON must be securely isolated and encrypted.
- **Data Retention:** `IngestedArticle` records, `ProcessedDigest` records, and their corresponding `DigestDeliveryAttempt` records must be physically deleted 7 days after `created_at` (subject to an operational cleanup schedule lag of up to 1 hour).
- **Global Caching:** `GlobalSource` records are shared entities. Their `url` must be unique to ensure fetch logic operates on a single record.
- **Feed Parser Deduplication Safeguard:** To prevent re-ingesting purged articles whose GUID or URL uniqueness checks have disappeared from the database, the feed parser must explicitly ignore and skip any feed items with a publication timestamp older than 7 days. For feed or web items that do not provide a publication timestamp, `published_at` must be stored as `NULL`; these items cannot use the publication-age check, and retention deletion instead relies on `created_at`, which is always the insertion time. Each `GlobalSource` must therefore maintain a persistent, bounded FIFO history in its `recent_items` JSON array, sized to hold at least 150 entries. This is guaranteed to exceed the maximum number of items returned in a single poll of standard RSS/Atom feeds. Each entry contains a URL hash and, when available, a GUID hash. During every polling cycle, any incoming item matching the applicable identifier in this history must be skipped. Newly accepted items must be appended, and the oldest entries removed only when the configured bound is exceeded. An item can be re-ingested only while it remains present in the active feed payload during a poll. Because the FIFO history exceeds the feed's maximum payload size, every active feed item remains in the deduplication history until it naturally falls out of the feed, preventing re-ingestion after its 7-day `IngestedArticle` row is purged.
- **Deduplication Partition Cross-Check:** To prevent duplicate inserts if a feed generator inconsistently includes and omits GUIDs for the same article across successive polling cycles, the Ingestion Service must perform an application-level URL cross-check against records from the same source only when the incoming article or the existing database record lacks a GUID. This handles both transitions: a GUID-bearing item followed by a no-GUID item, and a no-GUID item followed by a GUID-bearing item. If both the incoming article and an existing record have GUIDs, deduplication must rely strictly on GUID identity: the incoming item is skipped only when the GUIDs match. A shared URL with different GUIDs must not cause the incoming item to be skipped, because feeds may reuse one URL for distinct items such as podcast episodes or updates.
- **Redis Cache & Queue Payload Eviction:** To comply with the strict data retention SLA, any user news content cached in Redis must have a strict TTL configured (maximum 24 hours). BullMQ job queues must automatically evict finished or failed jobs and their payloads (`removeOnComplete: true` and `removeOnFail: { age: 86400 }`). Because BullMQ's age-based removal is lazy and runs only when subsequent jobs reach a finished state, the database cleanup worker must also run an explicit cleanup for every queue every 30 minutes, such as `queue.clean(86_400_000, 0, 'failed')` (the grace period is expressed in milliseconds). This deterministically removes failed jobs and payloads older than 24 hours even on inactive queues. Redis AOF files require the separate compaction policy described in the hosting documentation to erase the deleted payload bytes from disk.
