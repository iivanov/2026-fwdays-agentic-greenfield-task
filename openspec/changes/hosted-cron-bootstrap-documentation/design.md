## Context

The versioned scheduler migrations create three hosted cron jobs. Their command
uses a database runtime setting to resolve the project URL and uses the
scheduler secret for function authorization. That design avoids committing a
project identifier or secret, but a new hosted project can execute the local
fallback URL when the values are not entered after deployment. The observed
`pg_net` result was `Couldn't resolve host name`, which is consistent with a
hosted database trying to resolve the local `kong` hostname.

`curl` does not prove the cron configuration: it calls the hosted function URL
and supplies authorization directly. Deployment documentation must therefore
make the database-side bootstrap and the independent cron/HTTP evidence
mandatory before automatic reporting is considered available.

## Goals / Non-Goals

**Goals:**

- Provide one ordered, secret-safe procedure to bootstrap hosted cron after
  migration/function deployment.
- Explain the boundary between versioned scheduler SQL and per-project runtime
  values.
- Provide read-only verification and diagnostic queries that disclose neither
  the scheduler secret nor the service-role key.
- Require evidence that cron has a hosted target and that `pg_net` returns an
  HTTP result rather than a DNS error.

**Non-Goals:**

- Change the cron migration, Edge Function authorization, queue behavior, or
  provider configuration.
- Commit a project ref, a scheduler secret, a service-role key, or live
  provider state.
- Automate a production deploy or mutate a hosted project from CI.

## Decisions

### Keep project-specific values outside migrations

The guide will keep the public Supabase project URL and scheduler secret as
human-entered hosted database settings. Migrations remain portable and contain
only the setting names and the scheduler implementation. This satisfies the
public-repository secret boundary (`NFR-SEC-03`) and keeps one codebase usable
with a different Supabase project.

The guide will require `app.settings.supabase_url` and
`app.settings.scheduler_secret`. It will not instruct an operator to copy the
service-role key into a database setting: the deployed functions use their own
`SUPABASE_SERVICE_ROLE_KEY` secret internally, and cron authorizes with the
scheduler secret.

### Verify the complete scheduled path

The guide will add three read-only checks: registered cron jobs, cron execution
history, and `net._http_response` status/error fields. This separates a
missing/outdated schedule, function authorization failure, and DNS/HTTP
failure. It deliberately does not select response bodies, headers, or setting
values.

### Make cron bootstrap a deployment gate

The final deployment order and smoke checklist will include cron configuration
and a one-minute worker observation. `curl` remains an optional scheduler smoke
test, not evidence that automatic cron works.

## Risks / Trade-offs

- [Human bootstrap is still required] → The value cannot be generated by a
  portable migration; the guide identifies it as an explicit deployment step
  and gives a non-secret verification query.
- [An old schedule can remain] → The diagnostic query checks the installed job
  command for the local `kong` target and directs the operator to apply current
  migrations rather than editing `cron.job` directly.
- [Scheduled results are asynchronous] → The guide asks the operator to wait
  at least one worker interval before evaluating `pg_net` evidence.
- [Database settings and scheduler secret are privileged configuration] → Only
  an operator in the Supabase SQL Editor performs the bootstrap; values are
  never placed in Vercel browser variables, source files, logs, or test output.

## Migration Plan

1. Merge the documentation and OpenSpec artifacts only.
2. During a hosted deployment, apply migrations and deploy functions as usual.
3. Enter the hosted project URL and the already-configured scheduler secret in
   the SQL Editor using the documented commands.
4. Run the read-only cron/pg_net checks and correct an outdated migration before
   enabling users.

Rollback consists of restoring the prior documentation only; no database or
provider mutation is part of this change.

## Open Questions

None. A future deployment-automation change may introduce a secret-safe,
operator-approved bootstrap command, but it is outside this documentation
correction.
